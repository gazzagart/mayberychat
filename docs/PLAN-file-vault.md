# LetsYak File Vault — Feature Plan

## Overview

A personal cloud storage feature ("File Vault") integrated into LetsYak, where each user gets their own storage space and can share file links in chats, groups, and spaces. Files are stored once in the user's vault and shared by reference — one upload, many shares.

---

## 1. Architecture Decision

### Backend: MinIO (S3-compatible object storage) + thin Go API service

**Why MinIO over alternatives:**

| Factor | MinIO | Garage | SeaweedFS |
|--------|-------|--------|-----------|
| RAM footprint | 512MB–1GB | 256–512MB | 300MB–1GB |
| S3 compatibility | Full (reference impl) | Partial but sufficient | Partial |
| Dart `minio` package | ✅ v3.5.8, MIT, mature | ✅ (same package, S3-compat) | ❌ custom API |
| IAM / policies | Full (per-user keys, bucket policies) | Basic (admin API only) | Basic |
| Quota enforcement | Built-in bucket quotas | ❌ Must build | ❌ Must build |
| Docker | Official single-container | ✅ | ✅ |
| License | AGPL-3.0 | AGPL-3.0 | Apache-2.0 |
| Presigned URLs | ✅ Native | ✅ Native | ❌ |
| Production proven | Massive adoption | Smaller community | Large community |
| Web console | ✅ Built-in admin UI | ❌ | ✅ |

**MinIO wins because:**
1. **Built-in per-bucket quotas** — no custom quota code needed
2. **Full IAM** — per-user access keys with scoped policies, no custom auth layer
3. **Presigned URLs** — native support for time-limited share links with access control
4. **The `minio` Dart package** works directly with it (it was written for MinIO)
5. **Battle-tested** at massive scale, but runs fine in a single Docker container

**Garage is lighter on RAM** (~256MB vs ~512MB) but lacks built-in quotas and IAM, meaning you'd need to build a custom API wrapper for exactly the features MinIO gives you for free. The RAM difference is negligible given the overall stack (Synapse alone uses 500MB–1GB+).

### Why a thin API service (Go) between the app and MinIO?

The Flutter app should NOT talk directly to MinIO because:

1. **Auth integration** — We need to validate Matrix access tokens, not S3 credentials. The API service verifies the user's Matrix session and then issues scoped, short-lived S3 presigned URLs or temporary credentials.
2. **Share ACL enforcement** — When User A shares a file link with User B, the API service validates that User B is authorized (they're in the same room where the link was shared). MinIO alone can't check Matrix room membership.
3. **User provisioning** — When a new user first opens File Vault, the API service creates their MinIO bucket, sets quota, and generates their access policy.
4. **Quota management** — While MinIO has bucket quotas, the API service provides a user-friendly quota API (used/total, upgrade triggers) and handles payment-tier quota changes.
5. **Share tracking** — Records which files are shared in which rooms, so we can revoke access and show "shared with" metadata.

The app calls the Vault API for management operations (list files, create folder, get quota, share file, revoke share). File uploads/downloads use presigned URLs generated by the API, so the actual data flows directly between the app and MinIO — the API service is not a proxy for file content.

## 2. Data Model

### MinIO Structure

```
Bucket per user: vault-{matrix_user_id_hash}
  ├── documents/
  │   ├── report.pdf
  │   └── notes.txt
  ├── images/
  │   └── photo.jpg
  ├── projects/
  │   └── client-a/
  │       └── contract.pdf
  └── random-file.zip
```

### Vault API Database (PostgreSQL, shared with Synapse or separate)

```sql
-- User storage accounts
CREATE TABLE vault_users (
    matrix_user_id   TEXT PRIMARY KEY,      -- @user:homeserver.com
    bucket_name      TEXT NOT NULL UNIQUE,
    quota_bytes      BIGINT NOT NULL DEFAULT 524288000,  -- 500MB
    used_bytes       BIGINT NOT NULL DEFAULT 0,
    tier             TEXT NOT NULL DEFAULT 'free',       -- free, pro, enterprise
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Shared file links
CREATE TABLE vault_shares (
    share_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_user_id    TEXT NOT NULL REFERENCES vault_users(matrix_user_id),
    object_key       TEXT NOT NULL,          -- path in bucket
    file_name        TEXT NOT NULL,
    file_size        BIGINT NOT NULL,
    mime_type        TEXT,
    share_type       TEXT NOT NULL,          -- 'room', 'user', 'link'
    target_id        TEXT,                   -- room_id, user_id, or NULL for public link
    password_hash    TEXT,                   -- optional password protection
    expires_at       TIMESTAMPTZ,           -- optional expiry
    max_downloads    INT,                    -- optional download limit
    download_count   INT NOT NULL DEFAULT 0,
    is_revoked       BOOLEAN NOT NULL DEFAULT FALSE,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for looking up shares by room (for ACL checks)
CREATE INDEX idx_shares_target ON vault_shares(target_id) WHERE NOT is_revoked;
CREATE INDEX idx_shares_owner ON vault_shares(owner_user_id);
```

### Custom Matrix Event Type

When a user shares a file from their vault into a chat, the app sends a custom Matrix event:

```json
{
  "type": "com.letsyak.vault_file",
  "content": {
    "body": "📎 report.pdf (2.4 MB)",
    "msgtype": "com.letsyak.vault_file",
    "share_id": "a1b2c3d4-e5f6-...",
    "file_name": "report.pdf",
    "file_size": 2516582,
    "mime_type": "application/pdf",
    "vault_url": "https://vault.letsyak.com/share/a1b2c3d4-e5f6-...",
    "thumbnail_mxc": "mxc://homeserver/thumbnail_id",
    "owner": "@alice:homeserver.com"
  }
}
```

**Why a custom event type instead of `m.room.message` with custom `msgtype`?**

Using a fully custom event type (`com.letsyak.vault_file`) is cleaner because:
- It won't be rendered incorrectly by other Matrix clients (they'll show "unknown event" rather than a broken file message)
- The `body` field provides a human-readable fallback for clients that don't understand it
- It matches the pattern already used by Polls (`org.matrix.msc3381.poll.start`)

**Alternative considered:** Using `m.room.message` with `msgtype: "com.letsyak.vault_file"` would render as a text message fallback on other clients, which might be preferable. This is the approach used by `CuteEventContent`. **Recommendation: use this approach** — send as `m.room.message` with custom `msgtype` so other clients show the body text as a clickable link.

**Final decision — use `m.room.message`:**

```json
{
  "type": "m.room.message",
  "content": {
    "body": "📎 report.pdf (2.4 MB) — https://vault.letsyak.com/share/a1b2c3d4",
    "msgtype": "com.letsyak.vault_file",
    "share_id": "a1b2c3d4-e5f6-...",
    "file_name": "report.pdf",
    "file_size": 2516582,
    "mime_type": "application/pdf",
    "vault_url": "https://vault.letsyak.com/share/a1b2c3d4-e5f6-...",
    "thumbnail_mxc": "mxc://homeserver/thumbnail_id",
    "owner": "@alice:homeserver.com"
  }
}
```

This way:
- Other Matrix clients (Element, Nheko) show: `📎 report.pdf (2.4 MB) — https://vault.letsyak.com/share/a1b2c3d4` as a text message with a clickable link
- LetsYak renders a rich file card widget
- No timeline filtering issues (it's a standard `m.room.message`)

### Key Client Flows

**Upload a file to vault:**
1. User taps Vault in nav rail → `VaultPage` opens
2. User taps "+" / upload button → `FilePicker` opens (same as existing chat file picker)
3. File selected → `VaultUploadDialog` shows progress
4. App calls Vault API: `POST /api/v1/upload` with Matrix access token
5. API returns presigned PUT URL → app uploads directly to MinIO
6. API records file metadata, updates used_bytes

**Share a file from vault to chat:**
1. In chat, user taps "+" → "From Vault" → `VaultPage` opens in picker mode
2. User selects a file → `VaultShareDialog` opens (set expiry, password, etc.)
3. App calls Vault API: `POST /api/v1/shares` → returns `share_id`
4. App sends `m.room.message` event with `msgtype: "com.letsyak.vault_file"` into the room
5. Chat renders a `VaultFileCard` widget showing filename, size, download button

**Download a shared file (recipient):**
1. Recipient sees `VaultFileCard` in chat → taps download
2. App calls Vault API: `GET /api/v1/shares/{share_id}/download` with Matrix access token
3. API checks: is the user in the room where this share was posted? (Verifies via Matrix CS API)
4. If authorized → returns presigned GET URL → app downloads from MinIO
5. If unauthorized → 403 (file not accessible)

**Access from another Matrix client or web browser:**
1. The `body` text in the event contains the share URL: `https://vault.letsyak.com/share/a1b2c3d4`
2. Opening in browser → Vault API serves a download page
3. User authenticates via Matrix SSO or enters share password
4. Authorized → file downloads

## 4. Vault API Service (Go)

### Why Go?
- Tiny binary, minimal RAM (~20–50MB)
- Excellent S3 SDK (`minio-go`)
- Fast HTTP server (standard `net/http` or Chi router)
- Easy Docker deployment
- Already used in the Matrix ecosystem (Dendrite is written in Go)

### API Endpoints

```
POST   /api/v1/auth/provision       # Create user bucket + policy (called on first Vault open)
GET    /api/v1/quota                 # Get storage usage: { used, total, tier }
PUT    /api/v1/quota                 # Admin: update user quota/tier

GET    /api/v1/files?path=/          # List files in folder
POST   /api/v1/files/upload-url     # Get presigned PUT URL for upload
POST   /api/v1/files/download-url   # Get presigned GET URL for download
POST   /api/v1/files/folder          # Create folder
DELETE /api/v1/files?path=/x.pdf     # Delete file
POST   /api/v1/files/move            # Move/rename file

POST   /api/v1/shares                # Create share link
GET    /api/v1/shares/{id}           # Get share details
GET    /api/v1/shares/{id}/download  # Download shared file (validates access)
DELETE /api/v1/shares/{id}           # Revoke share
GET    /api/v1/shares/mine           # List all my active shares

GET    /share/{id}                   # Public web download page (HTML)
```

### Authentication

Every API request includes the Matrix access token in the `Authorization` header:
```
Authorization: Bearer <matrix_access_token>
```

The Vault API validates this by calling the Synapse admin API:
```
GET /_matrix/client/v3/account/whoami
Authorization: Bearer <token>
```

This returns `{ "user_id": "@alice:homeserver.com" }`, confirming the token is valid and identifying the user. No separate auth system needed.

### Share Access Control

When User B tries to download a file shared by User A:

1. API looks up the share record → gets `target_id` (room ID where it was shared)
2. API calls Synapse: `GET /_matrix/client/v3/rooms/{room_id}/joined_members` (using an admin token or the user's own token)
3. If User B is a member of the room → allow download
4. If not → 403

This means:
- If you're in the chat where a file was shared, you can download it
- If you leave the chat, you lose access
- If the share is revoked, nobody can download
- Optional: password-protected shares bypass room membership checks

## 5. Deployment

### Docker Compose Addition

```yaml
services:
  # ... existing: synapse, letsyak-web

  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: letsyak-admin
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD}
    volumes:
      - minio_data:/data
    ports:
      - "9000:9000"   # S3 API (internal, behind reverse proxy)
      - "9001:9001"   # Web console (admin only)
    restart: unless-stopped

  vault-api:
    image: letsyak-vault-api:latest
    build:
      context: ./vault-api
      dockerfile: Dockerfile
    environment:
      MINIO_ENDPOINT: minio:9000
      MINIO_ACCESS_KEY: letsyak-admin
      MINIO_SECRET_KEY: ${MINIO_ROOT_PASSWORD}
      SYNAPSE_URL: http://synapse:8008
      DATABASE_URL: postgres://user:pass@db:5432/vault
      VAULT_PUBLIC_URL: https://vault.yourdomain.com
    ports:
      - "8090:8090"   # Behind reverse proxy
    depends_on:
      - minio
      - db
    restart: unless-stopped

volumes:
  minio_data:
```

## 7. Risk Analysis

| Risk | Impact | Mitigation |
|------|--------|------------|
| Upstream merge conflicts | Low | All vault code is in new directories. Only 4 lines touch upstream files. |
| MinIO RAM overhead | Low | MinIO single-node uses ~512MB. Manageable alongside Synapse. |
| Large file uploads on mobile | Medium | Use presigned URLs (direct to MinIO). Add chunked upload for files >50MB. Progress bars via `dio` or `http` streaming. |
| Share link security | High | Presigned URLs expire (default 1 hour). Room membership validated on every download. Password support for extra-sensitive files. |
| Web CORS | Medium | Configure MinIO CORS policy for the app domain. Presigned URLs work cross-origin by default. |
| Matrix token expiry during upload | Low | Check token validity before generating presigned URL. Refresh if needed. |
| Offline access | Low | Files in vault are cloud-only. Consider optional local caching later (v2). |

## 9. Summary: Why This Approach

1. **Minimal upstream contamination** — 4 lines changed in existing files, everything else is new files in new directories
2. **No new Flutter dependencies** — uses existing `http` and `file_picker`
3. **Standard protocols** — S3 API for storage, Matrix access tokens for auth, presigned URLs for secure sharing
4. **Works across all platforms** — presigned URLs work on web, mobile, desktop — no platform-specific code needed
5. **Scales per-client** — each client deployment gets their own MinIO instance (or shared with bucket isolation)
6. **Monetizable** — quota tiers are a natural upsell
7. **Secure by default** — files are private. Sharing requires explicit action. Access is validated on every download via Matrix room membership.
8. **One file, many shares** — file lives in one bucket. Share links are just database records pointing to the same object.
